# AWS Encryption Blog Sample

A Terraform-based infrastructure project demonstrating a pattern that can support PCI DSS scope reduction using client-side Format-Preserving Encryption (FPE) with AWS Glue and an event-driven pipeline.

> **Disclaimer**: This is sample code accompanying an AWS blog post. It is provided for educational
> purposes only. Do not deploy to production without the additional hardening listed in
> [SECURITY.md](SECURITY.md), including (but not limited to) Lambda DLQs, KMS encryption of Lambda
> environment variables and CloudWatch Logs, X-Ray tracing, VPC Flow Logs, S3 access logging and
> lifecycle rules, Lambda code-signing, automatic rotation of the FPE key material, and
> `aws:SecureTransport` deny-policies on every S3 bucket and SQS queue. The code is not certified
> for any compliance regime, including PCI DSS.

## Overview

AWS does not natively support Format-Preserving Encryption for PCI data ingestion into Data Lakes/Lakehouses. This project showcases how to build a compliant data pipeline that:

- Ingests raw data into a quarantine zone
- Detects sensitive PCI data (credit card numbers, etc.) using Luhn algorithm, BIN lookup, and regex patterns
- Encrypts sensitive fields using Format-Preserving Encryption (FPE) while maintaining data format
- Moves encrypted data to a landing (raw) bucket for downstream Lakehouse consumption

## Architecture

```text
┌─────────────┐    ┌─────────────┐    ┌──────────┐    ┌────────────────┐    ┌─────────────┐
│ Quarantine  │───▶│ EventBridge │───▶│ SQS FIFO │───▶│ Lambda Trigger │───▶│    Step     │
│  S3 Bucket  │    │             │    │          │    │                │    │  Functions  │
└─────────────┘    └─────────────┘    └──────────┘    └────────────────┘    └──────┬──────┘
                                                                                   │
                                                                    ┌──────────────┼──────────────┐
                                                                    ▼              ▼              ▼
                                                              ┌───────────┐  ┌─────────────┐  ┌─────────┐
                                                              │ AWS Glue  │  │ Copy Lambda │  │ Landing │
                                                              │ (FPE Enc) │  │             │  │ Bucket  │
                                                              └───────────┘  └─────────────┘  └─────────┘
```

### Pipeline Flow

1. Files land in the **Amazon S3 Quarantine Bucket**
2. **Amazon EventBridge** detects `PutObject` events
3. **Amazon SQS FIFO** buffers events (ordering + deduplication)
4. **AWS Lambda Trigger** polls SQS and starts orchestration
5. **AWS Step Functions** orchestrates the pipeline:
   - Runs AWS Glue job (sensitive data detection + FPE encryption)
   - Glue calls the private Amazon API Gateway for FPE via the Vault Transform Service
   - Glue writes encrypted output back to quarantine under `/encrypted` suffix path
   - Triggers copy Lambda
6. **Copy Lambda** moves the encrypted file to the Landing Bucket, then deletes from quarantine

## Getting Started

### 1. Bootstrap the State Bucket

Run the bootstrap script once to create the S3 state bucket and generate `backend.hcl`:

```bash
./scripts/bootstrap.sh
```

This creates a bucket named `enc-blog-s3-tf-state-bucket-<your-account-id>` in `us-east-1` and writes a `backend.hcl` file with the resolved values.

### 2. Initialize Terraform

```bash
terraform init -backend-config=backend.hcl
```

### 3. Validate terraform

```bash
terraform validate
```

### 4. Plan terraform

```bash
terraform plan -var-file=tfvars/terraform.tfvars
```

### 5. Apply terraform

```bash
terraform apply -var-file=tfvars/terraform.tfvars
```

### 6. Seed FPE Key Material (one-time)

After the first deploy, invoke the secret generator Lambda to seed the FPE encryption keys into Secrets Manager:

```bash
aws lambda invoke --function-name enc-blog-lambda-secret-generator-function --payload '{}' /tmp/seed-output.json && cat /tmp/seed-output.json
```

You should see `{"status": "ok", "secretName": "enc-blog-sm-fpe-key-material-secret"}`. This only needs to be done once — subsequent deploys will skip if the secret is already seeded.

To rotate/regenerate the FPE key material:

```bash
aws lambda invoke --function-name enc-blog-lambda-secret-generator-function --payload '{"force": true}' /tmp/seed-output.json && cat /tmp/seed-output.json
```

### 7. Test the Pipeline

Upload a CSV file to the quarantine bucket under the `transaction/data/` prefix (required — EventBridge only triggers for this prefix):

```bash
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
aws s3 cp modules/ingestion-layer/assets/data/cards.csv s3://enc-blog-s3-quarantine-bucket-${ACCOUNT_ID}/transaction/data/cards.csv
```

The pipeline will automatically:

1. Detect the upload via EventBridge
2. Queue the event in SQS FIFO
3. Trigger the Lambda which starts Step Functions
4. Run the Glue job (detects sensitive data + FPE encrypts)
5. Copy the encrypted file to the landing bucket
6. Clean up quarantine

## Cleanup

**Warning:** Destroying this infrastructure permanently deletes all resources including S3 buckets with data, Amazon CloudWatch logs, and AWS Secrets Manager secrets containing FPE key material. Back up any data you need before proceeding.

1. Export any needed data from the landing bucket:

   ```bash
   aws s3 sync s3://enc-blog-s3-landing-bucket-$(aws sts get-caller-identity --query Account --output text) ./backup/
   ```

2. Destroy all Terraform-managed resources (takes ~10-15 min):

   ```bash
   terraform destroy -var-file=tfvars/terraform.tfvars
   ```

3. Delete the state bucket created by bootstrap (not Terraform-managed). Since versioning is enabled, you must delete all object versions first:

   ```bash
   BUCKET="enc-blog-s3-tf-state-bucket-$(aws sts get-caller-identity --query Account --output text)"

   # Delete all object versions
   aws s3api delete-objects --bucket $BUCKET --delete \
     "$(aws s3api list-object-versions --bucket $BUCKET --query '{Objects: Versions[].{Key:Key,VersionId:VersionId}}' --output json)" 2>/dev/null

   # Delete all delete markers
   aws s3api delete-objects --bucket $BUCKET --delete \
     "$(aws s3api list-object-versions --bucket $BUCKET --query '{Objects: DeleteMarkers[].{Key:Key,VersionId:VersionId}}' --output json)" 2>/dev/null

   # Remove the bucket
   aws s3 rb s3://$BUCKET
   ```

4. Verify in the AWS Console that no resources remain.

## How Encryption Works

The Glue job performs sensitive data detection and encryption in these steps:

1. **Read** — Loads the CSV file from quarantine into a Spark DataFrame
2. **Contract lookup** — Reads the treatment contract YAML from S3 to determine which columns are sensitive
3. **Detection** — For each sensitive column:
   - Extracts candidate card numbers via Spark-native regex
   - Validates against a BIN (Bank Identification Number) lookup table
   - Confirms with the Luhn algorithm
4. **Encryption** — Chunks detected values and sends batches to the private Vault Transform API (`POST /transform/encrypt`) which performs FF3-1 Format-Preserving Encryption
5. **Reconstruction** — Replaces original sensitive values with encrypted ones in the DataFrame
6. **Write** — Outputs the encrypted CSV (bz2 compressed) to quarantine under `/encrypted` suffix

The Copy Lambda then moves the encrypted file to the landing bucket and cleans up quarantine.

## Treatment Contract

The pipeline uses a YAML treatment contract (`assets/treatment-contract/cards-treatment-contract.yaml`) to define which columns contain sensitive data. To add new datasets:

1. Create a new YAML contract following the same schema
2. Update the `dataset` Glue job argument in `terraform.tfvars`
3. Place source files under the correct prefix in quarantine

## Expected Output

After a successful pipeline run:

- **Landing bucket** contains: `transaction/data/cards.csv.bz2` (encrypted, compressed)
- **Quarantine bucket** is cleaned up (original file + encrypted directory deleted)
- The encrypted file preserves the original data format — card numbers remain 16 digits but are FPE-encrypted

## Key Design Principles

- **Event-driven architecture** — loosely coupled services communicating via events
- **Least privilege IAM** — minimal permissions scoped to each resource
- **Resilience** — service and application-level retries with exponential backoff
- **PCI DSS scope reduction pattern** — encryption at rest and in transit, audit logging
- **Network isolation** — all compute in VPC private subnets, no internet access, all AWS calls via VPC endpoints
- **Fully parameterized** — zero hardcoded values; `terraform.tfvars` for resources, `backend.hcl` for state config
- **Naming convention** — `enc-blog-<service>-<purpose>-<resource-type>`
- **Region** — `us-east-1`

## Conclusion

This project demonstrates how Format-Preserving Encryption can be used to keep cardholder data out of downstream Lakehouse storage, using an event-driven, fully serverless architecture on AWS. The solution addresses a real compliance gap — encrypting PCI data while maintaining its format for downstream consumption — without requiring any internet access or exposing data outside the VPC.

The modular Terraform design makes it straightforward to adapt for other sensitive data types, add new datasets via treatment contracts, or integrate with existing data lake architectures.
